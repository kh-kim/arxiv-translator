Abstract

StarCoder와 같은 코드 대용량 언어 모델(Code LLMs)은 코드 관련 작업에서 탁월한 성능을 보여주었다. 그러나 대부분의 기존 모델은 명령어 미세 조정 없이 광범위한 원시 코드 데이터에 대해서만 사전 훈련된다. 본 논문에서는 WizardCoder 메서드를 코드 도메인에 적용하여 복잡한 명령어 미세 조정을 통해 Code LLMs에 권한을 부여하는 Evol-Instruct 메서드를 소개한다. 4개의 주요 코드 생성 벤치마크인 HumanEval, HumanEval+, MBPP 및 DS-1000에 대한 포괄적인 실험을 통해 모델의 탁월한 성능을 공개합니다. 다른 모든 오픈 소스 코드 LLM을 상당한 차이로 능가합니다. 게다가, 우리의 모델은 휴먼에벌과 휴먼에벌+에서 가장 큰 폐쇄형 LLM인 앤트로픽의 클로드와 구글의 바드를 능가한다. 코드, 모델 가중치 및 데이터는 https://github.com/nlpxucan/WizardLM에서 공개됩니다.

1 Introduction

최근 LLM(Large Language Models) [1, 2, 3, 4, 5, 6, 7, 8, 9]가 큰 주목을 받으며 인상적인 성공을 거두었다. 특히 OpenAI의 ChatGPT가 두드러진 사례로 눈에 띈다. 광범위한 양의 인터넷 데이터에 대한 광범위한 사전 교육을 활용하고 상세한 명령 데이터 [10]로 추가 미세 조정을 통해 이러한 모델은 다양한 작업에 걸쳐 최첨단(SOTA) 제로 샷 성능을 달성했다. 이러한 경향은 코드 이해와 생성의 영역에서도 관찰된다. 코드 관련 작업과 관련된 문제를 해결하기 위해 수많은 코드 LLMs[11, 12, 13, 14, 15, 16, 17, 18]가 제안되었다. 이러한 코드 LLM은 상당한 양의 코드 데이터를 사용하여 사전 훈련을 거쳐 다양한 코드 관련 작업을 탁월하게 수행할 수 있어 인상적인 성능을 보여줍니다.

사전 훈련 과정을 주로 강조하는 대부분의 이전 코드 LLM과 달리 코드 도메인에서 세밀한 명령어 튜닝에 대한 탐색은 제한적이었다. 명령어 튜닝의 도입은 처음에 다른 태스크들 [19, 20, 21, 22, 23, 24, 25]에 걸쳐 LM의 일반화 능력을 향상시키는 것을 목표로 했다. 예를 들어 OpenAI의 InstructGPT [10]는 사용자의 의도와 정렬을 보장하기 위해 명시적인 지침을 제공하기 위해 인간 주석자를 요청하는 것을 포함했다. 이와 유사하게 Alpaca [26]와 같은 최근 연구에서는 ChatGPT가 명령어 데이터를 생성하는 self-instruct [27] 방법을 사용하였다. Vicuna [28]는 ShareGPT.com에서 수집된 사용자 공유 대화를 활용했다. WizardLM [29]는 Evol-Instruct 메소드를 도입했는데, 이 메소드는 더 복잡하고 다양한 데이터 세트를 생성하기 위해 기존 명령 데이터를 진화시키는 것을 포함했다. 그러나 이러한 모든 접근법은 주로 일반 도메인에 초점을 맞추고 코드 도메인에 대한 구체적인 설계 고려 사항이 부족하다는 점에 주목할 필요가 있다.

Evol-Instruct 방법에 의해 동기화된 본 연구는 코드별 Evol-Instruct을 통해 복잡한 코드 명령어 데이터를 생성함으로써 SOTA 오픈 소스 Code LLM, StarCoder [11]의 성능을 향상시키는 것을 목표로 한다. 이를 위해 코드 관련 작업을 위해 특별히 조정된 진화적 프롬프트 프로세스에 몇 가지 적응을 수행했다. 이러한 수정은 진화 명령어를 정제하는 것, 진화 프롬프트의 형태를 단순화하는 것, 코드 디버깅 및 시공간 복잡성 제약들을 통합하는 것을 포함한다. 초기에는 기본 코드 명령 데이터인 Code Alpaca[30]를 진화시키기 위해 이 방법을 적용하였다. 그런 다음 새로 만든 코드 명령어 추적 훈련 세트를 사용하여 StarCoder의 미세 조정을 수행하고 WizardCoder을 얻는다.

4개의 코드 생성 벤치마크(HumanEval [31], HumanEval+ [32], MBPP [33], DS-100 WizardCoder이 다른 모든 오픈 소스 코드 LLMs보다 우수한 성능을 보여 SOTA(state-of-the-art) 성능을 달성했음을 보여준다. 특히, +22.3(57.3 vs. 35.0)의 증가로 패스@1 점수의 상당한 개선을 관찰한다. In HumanEval and +8.2 (51.8 vs. 43.6) (p<0.05). 놀랍게도, 훨씬 작은 크기에도 불구하고, 우리의 WizardCoder은 HumanEval과 HumanEval+의 합격률 면에서 Anthropic의 Claude와 Google의 Bard를 능가한다.

이 작품의 공헌은 다음과 같이 요약할 수 있다.

  • •

    우리는 Code WizardCoder의 적용을 통해 오픈 소스 Code LLM인 StarCoder의 성능을 향상시키는 Evol-Instruct을 소개한다.

  • •

    WizardCoder은 StarCoder, CodeGen, CodeGee, CodeT5+, InstructCodeT5+, StarCoder-GPTeacher 및 Instruct-Codegen-16B를 포함하여 코드 생성 측면에서 상당한 마진만큼 다른 모든 오픈 소스 Code LLMs를 능가합니다.

  • •

    WizardCoder은 크기가 상당히 작음에도 불구하고 Claude, Bard, PaLM, PaLM-2 및 LaMDA와 같은 가장 큰 폐쇄 소스 LLM과 비교하여 코드 생성에서 우수한 결과를 달성한다.

2 Related Work

Large Language Models.

Recently, LLMs have demonstrated remarkable achievements across a broad spectrum of tasks. Prominent tech companies have made significant strides in developing highly proficient LLMs. These include OpenAI’s GPT3&4 [1, 2], Google’s PaLM [3, 4], and Bard111https://bard.google.com/, DeepMind’s Chinchilla [5], and Gopher [6], as well as Anthropic’s Claude222https://www.anthropic.com/index/introducing-claude. However, it is important to note that these models are closed-source and can only be accessed through specific APIs or may not be accessible at all.

최근 LLM은 광범위한 작업에서 놀라운 성과를 보여주었다. 저명한 기술 회사들은 매우 능숙한 LLM을 개발하는 데 상당한 진전을 이루었다. OpenAI의 GPT3&4[1, 2], 구글의 PaLM[3, 4], Bard111https://bard.google.com/, DeepMind의 Chinchilla[5], 고퍼[6], Anthropic의 Claude222https://www.anthropic.com/index/introducing-claude 등이 그것이다. 그러나 이러한 모델은 폐쇄 소스이며 특정 API를 통해서만 액세스할 수 있거나 전혀 액세스할 수 없을 수 있다는 점에 유의하는 것이 중요합니다.

AI 커뮤니티는 모델 가중치가 공개적으로 공개되는 여러 오픈 소스 LLM이 출시되는 것을 목격했다. EleutherAI는 GPT-NeoX-20B [35]와 GPT-J-6B [36]에 기여했다. 구글은 UL2-20B [37]를 출시했다. 칭화대는 GLM-130B[7]를 도입했다. Meta has released OPT [9] and LLaMA [8]. 이러한 오픈 소스 모델은 귀중한 기여를 했지만 일반적으로 폐쇄 소스 모델과 동일한 수준의 성능을 나타내지 않는다는 점에 주목할 필요가 있다.

Large Language Models for Code.

최근 연구에서는 코드 이해 및 생성의 문제를 해결하기 위해 코드 관련 작업에 대해 상당한 수의 LLM을 도입했다. OpenAI는 Codex[16]와 Code-Davinci[38]를 공개했다. Google은 PaLM-Coder [3]를 제안했다. 그들은 HumanEval [31]와 MBPP [33]와 같은 인기 있는 코드 완성 벤치마크에서 뛰어난 성능을 보인다. 그러나 이러한 모델은 폐쇄 소스입니다.

한편, 이용 가능한 오픈 소스 코드 LLM은 여러 가지가 있다. Salesforce는 CodeGen [13], CodeT5 [17], CodeT5+ [18]를 도입하였다. 칭화대는 CodeGeeX[14]를, 빅코드 프로젝트는 StarCoder[11]를 개발했다. 이러한 모델은 코드 관련 작업에서 주목할 만한 발전을 보여주었다. 그러나 SOTA 폐쇄 소스 모델과 비교할 때 여전히 크게 뒤처진다. 명령어 미세 조정이 없는 앞서 언급한 모델과는 대조적으로, 본 연구는 Code Evol-Instruct을 사용하여 Code LLM을 추가로 훈련하면 성능을 크게 향상시킬 수 있음을 보여준다.

Instruction Fine-Tuning.

초기 단계에서 수업 미세 조정의 주요 목적은 LM의 교차 작업 일반화 능력을 향상시키는 것이었다. 이것은 공개 NLP 작업의 상당한 코퍼스로 LMs를 미세 조정함으로써 달성되었다. T5 [19]는 이 접근법을 탐구한 최초의 모델 중 하나였으며, 다수의 감독된 텍스트 대 텍스트 작업에 대한 훈련이었다. FLAN [20], ExT5 [22], T0 [23], UnifiedQA [25]와 같은 후속 작업은 LM의 전반적인 일반화 능력을 강화하기 위해 작업의 범위를 더욱 확장했다. 특히 ZeroPrompt [24]와 FLAN-T5 [21]는 훈련 파이프라인에 수천 개의 태스크를 통합하여 엔벨로프를 밀어붙였다. 이러한 연구 전반에 걸쳐 일관된 발견이 나타났는데, 다양한 NLP 작업 지침으로 LM을 미세 조정하면 새로운 작업에 적용할 때 상당한 성능 개선이 나타난다.

다양한 NLP 작업으로 LM을 미세 조정하면 유망한 결과를 보여주었지만 실제 사용자의 의도에 부합하지 못하는 경우가 많다. OpenAI는 다양한 형태와 광범위한 작업 유형을 포괄하는 인간 명령어의 대규모 코퍼스를 제공하기 위해 인간 주석자를 요청함으로써 다른 접근법을 추구했다. 이 데이터 세트를 기반으로 OpenAI는 GPT3[1] 모델을 훈련하여 InstructGPT[10]를 만들었으며, 이는 사용자의 입력과 더 잘 일치합니다. 이 발전 라인은 ChatGPT라고 알려진 인상적인 작업으로 이어졌습니다. 그러나 이러한 발전과 관련된 데이터 세트 및 모델 가중치는 공개적으로 사용할 수 없다는 점에 유의하는 것이 중요하다. Alpaca[26]는 Self-instruct 방법 [27]를 채택하여 ChatGPT를 활용하여 학습을 위한 데이터를 생성함으로써 다른 경로를 취한다. Vicuna [28]는 ShareGPT.com에서 수집된 사용자 공유 대화를 활용하여 모델을 학습합니다. WizardLM [29]는 Evol-Instruct 메서드를 소개하며, 이 메서드는 기존 명령 데이터를 진화하여 더 복잡하고 다양한 데이터 세트를 생성한다. 이러한 일반적인 명령 미세 조정 접근법과 달리, 우리의 WizardCoder은 코드 LLM의 도메인에서 특히 Evol-Instruct 방법을 성공적으로 적용했다.

3 Approach

이 섹션에서는 WizardCoder의 방법론적 세부 사항에 대해 자세히 설명한다. WizardLM에 이어 자체 명령을 사용하여 생성된 Code Alpaca를 진화하고 사전 훈련된 Code LLM StarCoder를 진화된 데이터로 미세 조정하기 위해 Evol-Instruct 방법을 적용한다.

3.1 Evol-Instruct Prompts for Code

WizardLM이 제안한 Evol-Instruct [29] 방법에서 영감을 받은 이 작업은 또한 코드 사전 훈련된 대형 모델의 미세 조정 효과를 향상시키기 위해 코드 명령을 더 복잡하게 만들려고 시도한다. Evol-Instruct를 코드의 영역에 적용하기 위해 진화 프롬프트를 다음과 같이 수정했습니다.

  1. 1.

    심화, 복잡한 입력 및 In-Breadth 진화를 제거하여 진화 지침을 간소화했습니다.

  2. 2.

    진화 프롬프트 템플릿을 통합하여 진화 프롬프트의 형태를 단순화했다.

  3. 3.

    코드 도메인의 특정 특성을 해결하기 위해 코드 디버깅과 코드 시공간 복잡도 제약의 두 가지 진화 명령어를 추가했다.

3.2 Training WizardCoder

다음 절차를 사용하여 WizardCoder을 학습합니다. 초기에는 StarCoder 15B [11]를 기반으로 하고 코드 명령어 추적 훈련 세트를 사용하여 미세 조정을 진행하며, 이는 Evol-Instruct을 통해 진화했다. 미세 조정을 위한 프롬프트 형식은 다음과 같이 요약됩니다.

훈련 데이터 세트를 구성하기 위해 코드 Alpaca333https://github.com/sahil280114/codealpaca이라는 20K 명령어 후속 데이터 세트로 초기화했다. 진화된 데이터를 생성하기 위해 20,000개의 샘플로 구성된 이 데이터 세트에 Evol-Instruct 기술을 반복적으로 사용한다. 데이터 진화의 각 라운드 후, 모든 이전 라운드에서 진화된 데이터를 원래 데이터 세트와 병합하여 Finetune StarCoder로 보내고 HumanEval [31]에서 pass@1 메트릭을 평가한다. 통과@1 메트릭의 감소를 관찰하면, 우리는 Evol-Instruct의 사용을 중단하고 가장 높은 통과@1을 갖는 모델을 궁극적인 모델로 선택할 것이다.

4 Experiment

이 섹션은 실험에서 기본 모델에 대한 포괄적인 개요를 제공하는 것으로 시작한다. 이후 4개의 코드 생성 벤치마크인 HumanEval [31], HumanEval+ [32], MBPP [33], DS-1000 [34]에 대한 모델의 성능을 제시한다.

4.1 Baselines

Closed-Source Models.

Multiple technology companies have successfully developed highly proficient LLMs while choosing not to publicly release them. These models are referred to as closed-source models. For our research, we incorporate a substantial number of these models as our baselines. Specifically, our baselines encompass the following: (i) OpenAI’s GPT3.5&4 [2], Code-Davinci-002 [38], Code-Cushman-001 [38], and Codex [16]; (ii) Google’s Bard, PaLM 2 [4], PaLM [3], and LaMDA [40]; (iii) Google DeepMind’s AlphaCode [12]; and (iv) Anthropic’s Claude.

여러 기술 회사가 고숙련 LLM을 성공적으로 개발하면서 공개적으로 출시하지 않기로 결정했습니다. 이러한 모델을 폐쇄 소스 모델이라고 합니다. 연구를 위해 이러한 모델의 상당수를 기준선으로 통합합니다. 구체적으로, 우리의 기준선은 (i) OpenAI의 GPT3.5&4 [2], Code-Davinci-002 [38], Code-Cushman-001 [38], Codex [16]; (ii) Google의 Bard, PaLM 2 [4], PaLM [3], LaMDA [40]; (iii) Google DeepMind의 AlphaCode [12]; 및 (iv) Anthropic의 Claude를 포함한다.

Open-Source Models.

4.2 Implementation Details

스타코더[11]는 우리의 기본 기초 모델 역할을 한다. 진화된 데이터 세트는 약 78k개의 샘플로 구성된다. 기본 모델을 미세 조정하기 위해 배치 크기 512, 시퀀스 길이 2048, 미세 조정 단계 200, 워밍업 단계 30, 학습 속도 2e-5, 코사인 학습 속도 스케줄러 및 fp16 혼합 정밀도를 포함한 특정 구성을 사용한다.

4.3 Evaluation on HumanEval, HumanEval+, and MBPP

HumanEval [31], HumanEval+ [32] 및 MBPP [33]는 Code LLMs 분야에서 광범위하게 활용되고 있는 벤치마크이다. 이러한 벤치마크는 코드 LLM에서 생성된 코드를 검증하기 위해 테스트 케이스를 사용하는 방대한 파이썬 프로그래밍 문제를 포함한다. HumanEval은 164개의 독창적인 프로그래밍 문제로 구성되어 있으며, 각 문제에 평균 9.6개의 테스트 케이스가 할당된다. LLM 합성 코드의 기능적 정확성에 대한 철저한 평가를 보장하기 위해 HumanEval+는 테스트 케이스 수를 크게 확장하여 문제당 평균 774.8개의 테스트 케이스로 확장한다. 반면에 MBPP는 500개의 테스트 프로그래밍 문제 세트를 제공하며 문제당 3개의 자동화된 테스트 케이스를 제공한다. 이러한 작업에 대한 프롬프트 형식은 다음과 같습니다.

Comparing with the Closed-Source Models.

GPT4, Claude 및 Bard와 같은 코드 생성을 위한 SOTA LLM은 주로 폐쇄 소스이다. 이러한 모델의 API에 대한 액세스 획득은 어려운 것으로 판명되었습니다. 본 연구에서는 LLM-Humaneval-Benchmarks [39]에서 HumanEval과 HumanEval+에 대한 점수를 검색하여 대안적인 접근법을 채택한다. 특히, 언급된 모든 모델은 단일 시도를 사용하여 각 문제에 대한 코드 솔루션을 생성하고 결과 통과율 백분율이 보고된다. 일관성을 유지하기 위해 그리디 디코딩을 사용하여 답변을 생성하여 동일한 실험 설정을 사용하고 제공된 평가 코드를 사용하여 WizardCoder을 평가한다. 이러한 표준화된 절차를 준수함으로써 기존 벤치마크에 대한 모델의 공정하고 유사한 평가를 보장하는 것을 목표로 한다.

1에 나타낸 바와 같이, 우리의 WizardCoder attains the third position in this benchmark, surpassing Claude-Plus (59.8 vs 53.0) (59.8 vs. 44.5). 특히, 우리 모델은 이러한 모델에 비해 훨씬 작은 크기를 나타낸다. 또한, WizardCoder은 명령어 미세 조정을 거치는 다른 오픈 소스 LLMs에 비해 뛰어난 성능을 보여 상당한 성능 마진을 보여준다.

Comparing with the Open-Source Models.

표 1에서 WizardCoder과 HumanEval 및 MBPP 벤치마크 상의 다른 오픈 소스 모델을 종합적으로 비교합니다. 1에 제시된 결과와 달리, 패스@1 점수를 추정하기 위해 각 문제에 대해 n개의 샘플을 생성함으로써 이전 연구 [31]에 요약된 접근법을 고수한다. 표 1에 제시된 결과는 우리의 WizardCoder이 모든 오픈 소스 모델에 비해 상당한 성능 이점을 나타냄을 분명히 보여준다.

1 및 표1의 실험 결과로부터 다음과 같은 결론을 얻을 수 있다:

  1. 1.

    WizardCoder은 훨씬 작음에도 불구하고 Claude, Bard, PaLM, PaLM-2 및 LaMDA를 포함한 가장 큰 폐쇄 소스 LLM을 능가합니다.

  2. 2.

    WizardCoder은 StarCoder, CodeGen, CodeGee 및 CodeT5+를 포함하여 모든 오픈 소스 Code LLMs를 큰 마진(+22.3 on HumanEval)으로 능가합니다.

  3. 3.

    WizardCoder은 InstructCodeT5+, StarCoder-GPTeacher 및 Instruct-Codegen-16B를 포함한 모든 오픈 소스 Code LLMs를 명령 미세 조정으로 상당히 능가합니다.

4.4 Evaluation on DS-1000

DS-1000 벤치마크 [34]는 7개의 라이브러리에 걸쳐 있는 1,000개의 별개의 데이터 과학 워크플로로 구성된다. 테스트 케이스에 대한 코드 생성의 성능을 평가하고 완료 및 삽입의 두 가지 평가 모드를 지원합니다. 실험에서는 지원하는 모델에 대한 삽입 점수만 보고한다. DS-1000 벤치마크는 Matplotlib (plt), NumPy (np), Pandas (pd), SciPy (scp), Scikit-Learn (sk), PyTorch (py), TensorFlow (tf)와 같은 라이브러리를 기반으로 문제를 더 분류한다. 스타코더와 동일한 프롬프트 형식을 따릅니다. 2에서는 전체 점수와 함께 각 라이브러리에 대한 pass@1(n=40) 결과를 제시한다. 이러한 결과에 기초하여, 우리의 결론은 WizardCoder이 DS-1000 벤치마크에서 데이터 과학 문제를 다룰 때 다른 모든 모델에 비해 상당한 우월성을 입증한다는 것이다. 이 관찰은 거의 모든 데이터 과학 라이브러리에서 적용된다.

4.5 Ablation Study

2는 데이터 진화 라운드 수의 영향을 조사하는 절제 연구를 제시한다. 진화된 데이터의 첫 번째 라운드에는 38k개의 샘플이 포함되어 있다. 두 번째 라운드에는 58k가 들어 있습니다. 세 번째 라운드는 78k입니다. 네 번째 라운드는 98k입니다. 일관성을 위해 모든 모델은 200단계로 미세 조정을 거칩니다. 결과는 인간성에 대한 가장 높은 패스@1 점수가 3회의 데이터 진화 후에 달성된다는 것을 보여준다. 이 관찰을 기반으로 3라운드 동안 진화한 데이터를 최종 데이터 세트로 선택한다.

5 Conclusion and Future Work

본 논문에서는 WizardCoder, Code Evol-Instruct fine-tuned Code LLM을 소개한다. 실험 결과는 WizardCoder이 HumanEval, HumanEval+, MBPP, DS-1000의 4가지 널리 알려진 코드 생성 벤치마크에서 기존의 모든 오픈 소스 코드 LLMs를 능가하는 SOTA 성능을 달성함을 보여준다. 또한 WizardCoder은 Anthropic's Claude, Google's Bard를 포함한 가장 큰 폐쇄형 LLMs에 비해 우수한 성능을 보인다.

Future Work.

WizardCoder은 그림 1에 묘사된 바와 같이 인상적인 코딩 성능을 보여주지만, 우리의 모델은 여전히 SOTA LLM인 GPT4에 크게 뒤떨어진다. 따라서, 향후 작업은 코드 Evol-Instruct 메서드의 향상을 우선시하여 우리 모델의 성능을 더욱 높일 것이다.

Broader Impact.

다른 LLM과 유사하게 WizardCoder도 비윤리적이거나 유해하거나 오판의 소지가 있는 정보를 생성할 수 있습니다. 따라서 윤리적, 사회적 함의를 해결하기 위한 향후 연구가 필요하다.
